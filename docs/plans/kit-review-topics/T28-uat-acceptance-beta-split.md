# T28 — UAT scope: agent-run acceptance vs task-based beta guides; "how do I?" is a design signal

**Category:** Convention (shipped standard + templates; reports + ui modules) · **Status:** **Decided (2026-07-10)** · **Issue:** #85 · **Related:** T5 (failure-driven diagnostics; T5.7 retired the role report templates), T17 (personas + journey-first specs), T26 (audience-first user-facing text), T24 (file naming), T13 (issue process), T11 (rationale lives once), T22 (grep-able IDs)

**Problem:** the reports module's `docs/uat/` was motivated by producing documents a non-technical, first-time beta tester could follow — but nothing shipped serves that audience. `UAT_SOURCE_OF_TRUTH.md` is entirely agent-facing acceptance verification (Playwright-backed E2E, criteria tables, evidence, precedence rules), and `docs/uat/` ships as a stub README. Two gaps compound it: (a) the acceptance doc the task-issue completion gate requires (`docs/uat/UAT_{{WORK_ITEM_PREFIX}}-NNN.md`) has **no template** — T5.7 retired the role *report* templates but the acceptance *spec* doc never got one, so agents author it freehand and its shape will drift across projects; (b) the originally imagined artifact — step-by-step tester scripts — would be self-defeating: scripted click-paths destroy the naïve first pass that makes human beta testing valuable. Guiding principle (Chris, 2026-07-10): *"User interfaces are like jokes. If you have to explain it, it's not very good."*

**Sub-decisions:**

- **T28.1 — Two artifacts, split by audience:** UAT is explicitly two things. **Acceptance verification** (agent-facing: criteria traceable to spec journey steps/EC-IDs, Playwright execution, evidence) — already standardized, unchanged. **Beta guides** (human-facing: hand-off docs for non-technical testers) — new. Playwright owns everything scripted; humans are reserved for what scripts can't give: naïve use, comprehension, discoverability.
- **T28.2 — Acceptance doc template:** add `ACCEPTANCE_DOC_TEMPLATE.md` (reports module `ai/TEMPLATES/`) closing the freehand gap — acceptance-criteria table citing journey step numbers and edge-case IDs, edge-case matrix, outcome columns; the shape `UAT_SOURCE_OF_TRUTH.md`'s completion gate already implies.
- **T28.3 — Beta guides are goals, not steps:** add `BETA_GUIDE_TEMPLATE.md`. Addressed to a persona by name (`docs/PERSONAS.md`), written under T26 rules. Each task is a scenario ("you just got back from a home visit and need to record it"), a starting point, and a done-condition stated from the user's perspective — **never a click-path** (steps would explain the joke and hide discoverability failures). Per task, three feedback prompts: did you finish? where did you pause or guess? what did you expect that didn't happen? Plus a standing invitation to wander off-script, and a header with prerequisites, time estimate, and where to send feedback.
- **T28.4 — Lifecycle:** guides are **per feature, derived from the feature spec's journey layer** (the journey with the steps deleted, leaving goals). Agent drafts at feature-complete; a human reviews before testers see it (it's user-facing text). Release cut is the refresh point — `CHANGELOG.md [Unreleased]` is the delta of guides to create/update for a beta round. Optional early use: drafting the guide *before* the UI is finalized is a free design test — a task that can't be stated without explaining steps fails the joke test pre-tester.
- **T28.5 — Sequencing:** testers receive guides only after acceptance verification and the E2E smoke are green on a build they can touch. Rationale: a tester's naïve first pass through a flow is nonrenewable — never spend it on failures Playwright would have caught.
- **T28.6 — "How do I…?" routes to design:** a how-question from a user or beta tester is a **design finding, not a documentation request**. Default routing: file a UX/design issue against the flow; write help text only if a redesign is explicitly rejected. Single home for the rationale (T11): `UI_STANDARD.md`; one-line pointers in the UAT standard's feedback handling and the beta guide template.
- **T28.7 — Location & naming:** beta guides live in **`docs/uat/beta/`** (reports module). Keeps them inside the module's owned tree (scaffold trigger + manifest entries stay with reports), separates audience by folder, and leaves every existing `docs/uat/UAT_*-NNN.md` reference (task-issue template, `/qa`, source-of-truth precedence) untouched. Guides are ID-anchored to their feature spec: `BETA-<DOMAIN>-NNN-<slug>.md` mirrors `SPEC-<DOMAIN>-NNN` (T24.3/T22). Considered and rejected: `{{DOCS_SOURCE_OF_TRUTH}}` (guides are versioned test artifacts tied to beta rounds, not durable product docs — and the docs source of truth may live outside the repo) and a new top-level `docs/beta/` (splits module ownership for no audience gain).

New shipped files (two templates + `docs/uat/beta/README.md`) → `template/manifest.yml` entries in the same PR (T23.2).

**Decision (Chris, 2026-07-10 — "It's decided lets record and implement this"):** T28.1–T28.7 adopted as drafted; implementation tracked as issue #85.
